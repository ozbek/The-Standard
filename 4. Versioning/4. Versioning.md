# 4 Versioning

## 4.0 Introduction
Versioning is a discipline that governs how systems signal change without breaking the consumers that depend on them. In any system that evolves over time, changes to models, services, or APIs are inevitable. The Standard treats versioning not as an afterthought but as a first-class concern — one that must be explicit, structured, and safe for every consumer that relies on the system.

Versioning in The Standard covers four primary areas: release versioning, file versioning, API versioning, and deprecation signaling. Each of these areas works together to ensure that no change is hidden, no consumer is surprised, and no version increment is ambiguous.

## 4.0.0 Purpose
The primary purpose of versioning is to communicate the nature of a change, not merely that a change occurred. A version number in The Standard tells a consumer exactly what kind of change happened and what they should expect as a result. A model change is structurally different from a service behavior change, and both are different from a bug fix. Collapsing these distinctions into a single undifferentiated increment leaves consumers guessing about what actually changed and what they need to do about it.

### 4.0.0.0 Non-Destructive by Design
The most important principle of versioning in The Standard is that versioning must never destroy what already exists. When a model or service changes, a new versioned file is introduced alongside the original. The original remains in place, untouched, so that existing consumers continue to function without modification. This non-destructive approach means that introducing a new version is always additive — the earlier version remains available until an explicit deprecation and removal strategy is put in place outside the introduction of the new version.

## 4.0.1 Types of Versioning
Versioning in The Standard applies at three distinct levels: release versioning, file versioning, and API versioning. Let's discuss each of those briefly.

### 4.0.1.0 Release Versioning
Release versioning governs how a system communicates what kind of change has been made at the release level. The Standard uses a strict four-segment format for all release versions:

```text
v1.2.3.4
```

Each segment carries a fixed and non-negotiable meaning:

| Segment | Meaning |
|---|---|
| 1 | Model change |
| 2 | Service or routine change |
| 3 | Bug fix or configuration change |
| 4 | Automated build version |

When multiple change types occur in the same release, the highest-order change wins and all lower segments reset to zero. Given a current version of `v1.2.3.4`, the expected next versions for each change type are:

- model change → `v2.0.0.0`
- service or routine change → `v1.3.0.0`
- bug fix or configuration change → `v1.2.4.0`
- automated build of the same code → `v1.2.3.5`
- model + service change together → `v2.0.0.0`

The version number must describe what kind of change happened, not merely that a change happened. That means a model change is never hidden inside the service segment, a service change is never hidden inside the bug or configuration segment, and a build increment is never used to represent a code change.

### 4.0.1.1 File Versioning
File versioning governs how individual files on disk reflect the evolution of a model or service. A file that has no version folder is treated as version zero — V0 is implied by its absence. This keeps the starting shape of a system clean and uncluttered while still establishing a logical baseline that all later versions build upon.

#### 4.0.1.1.0 Model Versioning
When a model changes, a new versioned file is introduced in a nested subfolder that makes the version number explicit in both the folder name and the type name:

```text
Models/Foundations/Students/V1/StudentV1.cs
```

The same principle applies to services that depend on that model. A changed model produces a new versioned service file alongside the original, not in place of it:

```text
Services/Foundations/Students/V1/StudentV1Service.cs
```

Exceptions related to the new model version follow the same nesting convention and live beneath the corresponding versioned model path:

```text
Models/Foundations/Students/V1/Exceptions/
```

#### 4.0.1.1.1 Service Behavior Versioning
When the behavior of a service changes without any change to the underlying model, the service file itself must be versioned to reflect that behavioral evolution. A behavior change is any modification to how the service operates — for example, changing validation rules (such as increasing a maximum field length from 20 to 30 characters), making a previously optional field required, or altering business logic.

If the model is still at V0 and only the service behavior changes, the versioned service file reflects the behavior version:

```text
Services/Foundations/Students/StudentServiceV1.cs
```

If the model has already been versioned to V1 and the behavior of that V1 service changes, the service file reflects both the model version and the behavior version:

```text
Services/Foundations/Students/V1/StudentV1ServiceV1.cs
```

The behavior version always appears as a suffix after the service name. This makes both the model version and the behavior version visible and unambiguous.

#### 4.0.1.1.2 Combined Model and Behavior Changes
When both the model and the service behavior change together in the same release, only the model version increments. The behavior version resets to V0, implied by its absence from the file name. This follows the same reset principle used in release versioning — the highest-order change wins and all lower segments reset. A new model always means new service behavior by definition, so the behavior version returns to its baseline.

For example, if both the model and service behavior change together from `StudentV1ServiceV1` to a new model version, the resulting file is:

```text
Services/Foundations/Students/V2/StudentV2Service.cs
```

The service behavior version is once again implied as V0, and the cycle can repeat as further behavior changes are introduced for the V2 model.

#### 4.0.1.1.3 Naming Conventions
The naming convention for versioned files is strict and reflects both the model and behavior version when applicable:

- A versioned model is named `{Entity}V{n}.cs`
- A versioned service with model version only is named `{Entity}V{n}Service.cs`
- A versioned service with behavior version only is named `{Entity}ServiceV{n}.cs`
- A versioned service with both model and behavior versions is named `{Entity}V{m}ServiceV{n}.cs`

This makes the version visible at every level — in the folder structure, in the file name, and in the type name itself.

### 4.0.1.2 API Versioning
API versioning governs how routes communicate the model and behavior version to the consumers of a system. The initial API is exposed without any version prefix, keeping the route clean and straightforward:

```text
api/Students
```

This is the V0 route. When a model change occurs, the route must surface the model version so that consumers understand they are interacting with a structurally different shape:

```text
api/V1/Students
```

When the service behavior changes for an already-versioned model, the route surfaces both the model version and the behavior version together as a pairing:

```text
api/V1.1/Students
```

In this shape, the number before the dot identifies the model version and the number after the dot identifies the behavior version. This pairing gives consumers a precise signal about both the shape of the data they will receive and the behavior they should expect from the service. If only the behavior changes but the model remains at version zero, the route becomes `api/V0.1/Students` — the model version is still zero, but the behavior version has incremented to one.

## 4.0.2 Deprecation
Deprecation signals that a version of an API or piece of code is no longer the recommended path forward and will eventually be removed. For deprecation to be useful, it must be visible. Consumers must be able to see that an API is deprecated, understand when it is expected to sunset, and know where to go for migration guidance. They must also be given enough time to act before the deprecated version disappears.

For APIs, a valid approach is to surface deprecation metadata through response headers using an attribute applied directly to the controller action:

```csharp
[ApiController]
[Route("api/[controller]")]
public class SampleController : ControllerBase
{
    [HttpGet]
    [DeprecatedApi(
        Sunset = "2024-12-31",
        Warning = "This API is deprecated. Please migrate to v2.",
        Link = "https://example.com/deprecation-info")]
    public IActionResult GetSampleData()
    {
        return Ok(new { message = "Sample data" });
    }
}
```

The [`Attrify`](https://www.nuget.org/packages/Attrify) NuGet library provides the `DeprecatedApi` attribute and handles the emission of the appropriate deprecation headers automatically, making the lifecycle of an API a compile-time concern rather than a runtime configuration task.

For code that is no longer maintained or supported, the standard `[Obsolete]` attribute makes deprecation visible to maintainers and consumers at compile time:

```csharp
[Obsolete("This version is deprecated and no longer maintained.")]
```

## 4.0.3 Capabilities
Capabilities are a discovery mechanism that allows consumers to understand what a system or provider supports before they ever attempt to invoke it. Without a capabilities surface, a consumer has no choice but to call an operation and react to whatever failure or exception comes back. With one, the consumer can check first and decide safely. This aligns directly with the broader Standard principle of avoiding invalid operations rather than reacting to them after the fact.

In a provider-based architecture, each concrete provider exposes a capabilities property that enumerates the operations it supports for each resource. The consumer queries this property before invoking any operation, making the interaction deterministic and safe:

```csharp
IStudentProvider provider = new StudentProvider();

if (!provider.SupportsOperation("Student", "Add"))
{
    return;
}

await provider.Students.AddAsync(new Student());
```

Capabilities may also be surfaced through Swagger, middleware-based metadata, bespoke capabilities endpoints, or provider-specific capabilities endpoints, depending on the architecture of the system. The goal in every case is the same: turn runtime uncertainty into a deterministic check that the consumer can perform before committing to a call.

## 4.0.4 Examples
Let's take file versioning as a concrete example of how versioning works in practice across a full model-to-API chain.

Suppose the initial shape of a student system looks like this:

```text
Models/Foundations/Students/Student.cs
Services/Foundations/Students/StudentService.cs
api/Students
```

When the `Student` model changes — for instance, a new required property is introduced — a new versioned model is created alongside the original. The original is left untouched:

```text
Models/Foundations/Students/V1/StudentV1.cs
```

A new versioned service is introduced to serve that new model:

```text
Services/Foundations/Students/V1/StudentV1Service.cs
```

And the API route surfaces the model version so that consumers can choose which version they integrate with:

```text
api/V1/Students
```

The original files and the original route remain exactly where they are. Consumers of the V0 version are completely unaffected. At the release level, the version increments from, say, `v1.2.3.4` to `v2.0.0.0` to reflect that a model change has occurred — and that increment alone tells every consumer precisely what kind of change to expect.

## 4.0.5 Summary
In summary, versioning in The Standard is a structured, non-destructive discipline that communicates the nature of every change clearly and without ambiguity. Release versions describe what kind of change happened. File versions preserve all earlier implementations alongside their successors. API routes surface the model and behavior version as an explicit pairing. Deprecation signals give consumers the visibility and time they need to migrate safely. And capabilities turn runtime uncertainty into a deterministic check before an operation is ever invoked.
